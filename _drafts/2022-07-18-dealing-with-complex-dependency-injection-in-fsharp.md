Это перевод статьи Бартош Сыпитковски ["Работа со сложным внедрением зависимостей в F#"](https://bartoszsypytkowski.com/dealing-with-complex-dependency-injection-in-f/).

Сегодня мы рассмотрим различные способы инкапсуляции зависимостей и передачи их между функциями используя техники функционального программирования которые могут реализованы на F#.

<!--more-->

Управление зависимостями в объектно-ориентированных языках к 2020 году в значительной степени проблема с однобоким решением - внедрение зависимостей победило, люди используют специальные фреймворки для управления зависимостями которые на 99,9% работают используя рефлексию. Конечно, теперь эти фреймворки тоже нужно изучить, потенциально неправильно сконфигурировать, что приведёт к ошибкам рантайма или, возможно, даже обнаружить, что не каждая проблема - stateless сервис, но это все равно лучше (?), чем то, что было в прошлом, да и в любом случае, что мы можем ещё сделать?

В мире функционального программирования нет единого однозначного решения или подхода - было предложено много различных вариантов, в зависимости от возможностей языка и компилятора. И поскольку практически все функциональные языки предлагают паттерн, известный как *частичная реализация*, в течение многих лет это был наиболее распространенный ответ на проблему управления зависимостями.

Короче говоря, мы говорим о внедрении зависимостей с помощью параметра функции:

``` fsharp
// foo требует зависимости bar и baz для обработки входящего запроса
let foo bar baz request = ???
// зависимости предоставляются через частичную реализацию
let wired = foo dependency1 dependency2
// теперь wired может обрабатывать запросы напрямую, без вызова зависимостей каждый раз 
let response = wired request
```

Очень простой способ, который не требует рефлексии или специальной библиотеки. Однако при таком подходе появляется несколько проблемных моментов, особенно заметных по мере роста и усложнения кодовой базы. Последние решения, популяризированные такими библиотеками, как Scala ZIO или Haskell's Polysemy, бросают вызов этому подходу.

## Частичная реализация

Есть некоторые вопросы, на которые частичная реализация не даёт чёткого ответа.

Представьте, что вы используете набор методов, которые тесно связаны и инкапсулированы в один объект, например запросы в базу данных или различные методы логирования (debug / info / warning / error). Теперь, учитывая, что функция потенциально может использовать более одного из этих методов, как нужно передавать аргументы? Есть несколько подходов:

1. Путь пуриста-функциональщика - передавать каждую зависимость как отдельный параметр функции, например:

   ``` fsharp
   let doSmth logError logInfo = ??
   ```

   Хотя такой подход и позволяет точно описать, что использует функция, но он приводит к взрывному росту количества параметров функции. Кроме того, каждый раз, когда необходимо создать новую функцию в существующем коде, то приходится частично реализовывать её перед вызовом.

2. Описывать зависимости используя алгебраические типы данных и внедрять на их место функции, например:

    ``` fsharp
    let doSmth (log: LogEvent -> unit) = ??
    ```

    Такие зависимости легко мокировать, поскольку не нужно реализовывать всё, а только соответствие функции типу в случаях, которые важны для конкретного теста. Также, такой подход уменьшает количество схожих передаваемых параметров, но, с другой стороны, может затруднить понимание, особенно во время отладки. Также иногда такое решение может вызвать снижение производительности.

3. Вернуться к объектам/интерфейсам и передавать их в качестве параметров, например:

   ``` fsharp
   let doSmth (logger: ILogger) = ??
   ```

   Хотя интерфейсы могут упростить дерево зависимостей, не всегда очевидно, когда их использовать. К тому же мокирование таких зависимостей сложнее.

Это довольно распространенные подходы, которые я встречал в своей жизни, и каждый из них имеет свои преимущества и недостатки. Какой из них использовать? Хороший вопрос, так как на практике в проектах с достаточно старой кодовой базой обычно можно встретить два или даже все три вперемешку. Со временем, это может привести к путанице и неясности.

Что еще хуже, ни один из этих подходов на самом деле не решает проблему управления зависимостями - всё, что они делают, это просто пытаются уменьшить проблему. В итоге, придется вручную, путем частичной реализации, связывать десятки функций и вручную управлять всеми зависимостями между ними.

### Пример

Чтобы лучше понять, будем использовать довольно простой пример - изменение пароля пользователя:

``` fsharp
let fetchUser (db: IDbConnection) userId = 
    db.QueryFirstAsync(Sql.FetchUser, {| userId = userid | })
    
let updateUser (db: IDbConnection) user = db.ExecuteAsync(Sql.UpdateUser, user)

let changePass (logger: ILogger) fetch update = fun req -> task {
    let! user = fetch req.UserId
    if user.Hash = bcrypt user.Salt req.OldPass then
        let salt = generateSalt ()
        let user' = { user with Salt = salt; Hash = bcrypt salt req.NewPass }
        do! update user'
        logger.LogInformation "Password change: user %i" user.Id
        return Ok ()
    else 
        logger.LogError "Password change unauthorized: user %i" user.Id
        return Error "Old password is invalid"
}
```

В этом коротком фрагменте кода есть несколько зависимостей. Но сколько их в действительности? Рассмотрим подробнее:

- Количество параметров `changePass` равно 3, но в зависимости от решений в будущем, их может стать 4 (если передавать `LogInformation` и `LogError` отдельно) или 2 (если выделить `fetch` и `update` в интерфейс).
- Нетрудно представить, что в будущем функция хеширования `bcrypt` может стать конфигурируемой - возможно, даже для каждого пользователя. Для этого может потребоваться отдельный параметр.
- Может быть, помимо логирования, понадобиться отдельный механизм телеметрии для подсчета количества входящих запросов или количества неудачных валидаций пароля? Это означает еще один параметр.
- Генерация соли - это псевдослучайный процесс. Если необходимо, чтобы функция была детерминированной, вероятно, следует также параметризовать ее по явно переданному случайному значению.

Как видно, то, что вначале казалось простой задачей, может быстро стать серьёзной проблемой. С ростом количества аргументов, код для связывания функций между собой становится более гадким. Довольно распространенный паттерн заключается в том, чтобы спрятать всю эту гадость под ковром a.k.a. **корень композиции**. Однако не обязательно всё должно быть именно так.

Ниже, поэтапно, от базовых принципов до монадических привязок, рассмотрен другой подход к работе с зависимостями, вдохновленный библиотекой Scala - [ZIO](https://medium.com/@pascal.mengelt/what-are-the-benefits-of-the-zio-modules-with-zlayers-3bf6cc064a9b).

## Управление зависимостями без частичной реализации

Рассмотрим вначале то, как будет выглядеть приведенный выше код по итогам этого параграфа:

``` fsharp
let changePass env = fun req -> task {
    let! user = Db.fetchUser env req.UserId
    if user.Hash = bcrypt user.Salt req.OldPass then
        let salt = Random.bytes env 32
        do! Db.updateUser env { user with Salt = salt; Hash = bcrypt salt req.NewPass }
        Log.info env "Changed password for user %i" user.Id
        return Ok ()
    else 
        Log.error env "Password change unauthorized: user %i" user.Id
        return Error "Old password is invalid"
}
```

Можно заметить, что все параметры были заменены каким-то одним загадочным параметром `env`. К нему мы ещё вернёмся. Также были инкапсулированы функции `fetch`, `update`, `logger` и `generateSalt` в соответствующие модули `Db`, `Log` и `Random`. Начнём с определения `Log`:

``` fsharp
[<Interface>]
type ILogger =
    abstract Debug: string -> unit
    abstract Error: string -> unit 

[<Interface>] type ILog = abstract Logger: ILogger

module Log =
    // env: 'a (requires :> ILog ) -> fmt: Printf.StringFormat<'b,unit> -> 'b
    let debug (env: #ILog) fmt = Printf.kprintf env.Logger.Debug fmt
    //  env: 'a (requires :> ILog ) -> fmt: Printf.StringFormat<'b,unit> -> 'b
    let error (env: #ILog) fmt = Printf.kprintf env.Logger.Error fmt
```

Теперь можно сказать кое-что еще о `env`. Секрет заключается в [гибком типе](https://docs.microsoft.com/en-us/dotnet/fsharp/language-reference/flexible-types) `#ILog`. Такое объявление обозначает, что `env` может быть любым типом, реализующим интерфейс `ILog`. Вскоре будет показано, насколько этот подход удобен для компоновки, но перед этим нужно определить еще один модуль:

``` fsharp
[<Interface>]
type IDatabase =
    abstract Query: string * 'i -> Task<'o>
    abstract Execute: string * 'i -> Task

[<Interface>] type IDb = abstract Database: IDatabase

module Db = 
    // env: 'a (requires :> IDb ) -> userId: 'b -> Task<'c>
    let fetchUser (env: #IDb) userId = 
        env.Database.Query(Sql.FetchUser, {| userId = userId |})
    // env: 'a (requires :> IDb ) -> user: 'b -> Task
    let updateUser (env: #IDb) user = 
        env.Database.Execute(Sql.UpdateUser, user)
```

Что произойдет, если будут использованы функции как из модуля `Log`, так и из модуля `Db`? Как оказалось, компилятор F# может правильно выводить ограничения обобщений для таких типов. Результирующее ограничение типа для `env` будет объединением ограничений - точно так же, как объединение множеств. Это означает, что компилятор обрабатывает дубликаты - все ограничения других функций, использующих `env` в своей области видимости:

``` fsharp
let foo env = // env :> IDb and env :> ILog
    let user = Db.fetchUser env 123 // env :> IDb
    Log.debug env "User: %A" user   // env :> ILog
```

Теперь о том, почему были использованы два разных интерфейса `ILog` и `ILogger` вместо того, чтобы реализовать `ILogger` напрямую? Этот подход более практичный и позволит изолировать функции конкретных модулей, а не добавлять их напрямую в `env`, например:

``` fsharp
module Log =
    let live : ILogger = ?? // create logger interface

[<Struct>]
type AppEnv = 
    interface ILog with member _.Logger = Log.live
    interface IDb with member _.Database = Db.live connectionString

foo (AppEnv())
```

Невозможно передать в `foo` конкретную реализацию `ILog` или `IDb`, потому что `env` должен иметь тип реализующий оба интерфеса. При этом, для поддержания инкапсуляции, модуль `Log` не должен знать о существовании или реализации интерфейса `IDb`. То же самое верно и для модуля `Db`. Однако, можно написать функцию `live` которая реализует `ILogger`, и инкапсулирует возможности, требуемые модулем `Log`. Таким образом, не нужно знать подробности `ILogger` при определении `AppEnv`.

Плюсы такого подхода:

- Нужно передавать только `env` вместо потенциально неограниченного списка параметров. Поскольку он всегда один, его легче сделать обобщённым и скомпоновать в него другие функции.

- В отличие от DI-фреймворков основанных на рефлексии - типобезопасность по прежнему сохраняется и проверяется компилятором. Если тип параметра `env` не будет реализовывать интерфейс, требуемый где-то в цепочке вызовов, наш код просто не будет компилироваться.

- Unit-тестирование по-прежнему довольно простое - каждая функция определяет только те ограничения обобщений, которые она использует в своем собственном дереве вызовов, а не все ограничения, требуемые приложением.

- Новые зависимости добавляются неявно - если код использует модуль, который требует дополнительных функций, то он будет автоматически выведен компилятором и отображен в определении типа параметра `env`. Нет необходимости добавлять новые параметры функции или передавать новый аргумент. Кроме того, в отличие от объектно-ориентированных контейнеров IoC, нет необходимости добавлять для новых зависимостей поля классов и аргументы конструкторов.

- Помогает в определении того, что должно быть зависимостью - меньше размышлений о том, «должна ли это быть функция или интерфейс?» или «если эти две функции соответствуют одному и тому же функционалу, должны ли они передаваться по отдельности?».

- Не налагает ограничений на библиотеки и фреймворки.

На этом можно было бы остановиться, т.к., по моему мнению, такой подход уже хорош и полезен для большинства случаев. Но можно и попробовать пойти дальше. Код представленный выше требует постоянно передавать `env`. Можно ли что-нибудь с этим сделать? Оказывается, что да, можно.

## Монада Reader

Перед тем как продолжить: **то, что сейчас будет рассмотрено, менее полезно с точки зрения текущего состояния экосистемы F# по причинам, которые будут упомянуты позже.**

Паттерн, который будет использован здесь, известен как [Монада Reader](https://fsharpforfunandprofit.com/posts/elevated-world-6/). Хотя этот паттерн и полезен в определенных ситуациях, он не получил широкого распространения. На мой взгляд, это из-за названия, которое звучит одновременно бессмысленно и пугающе в ушах многих разработчиков.

Остальная часть этого поста будет посвящена введению в этот паттерн на F#, однако сосредоточена исключительно на проблеме управления зависимостями - другие аспекты монад будут проигнорированы.

Вернёмся к ранее определённому типу параметра `env`, но теперь поместим его непосредственно в другой тип, который назовем `Effect`. Поскольку уже было упомянуто, что в рассматриваемом паттерне есть М-слово, то можно с уверенностью предположить, что логика нашего обработчика будет определена как ленивая последовательность, которая должна быть выполнена (звучит почти как `async` / `await`). Добавим немного синтаксического сахара, используя выражения вычисления `effect { ... }` которое будет возвращать тип `Effect`:

``` fsharp
[<Struct>] type Effect<'env, 'out> = Effect of ('env -> 'out)
```

Где:

- `'env` - тип, о котором говорили выше.
- `'out` - определяет тип возвращаемого значения нашего эффекта.

В итоге, обработчик для запросов будет выглядеть следующим образом:

``` fsharp
let changePass req = effect {
    let! user = Db.fetchUser req.UserId
    if user.Hash = bcrypt user.Salt req.OldPass then
        let! salt = Random.bytes 32
        do! Db.updateUser { user with Salt = salt; Hash = bcrypt salt req.NewPass }
        do! Log.info "Changed password for user %i" user.Id
        return Ok ()
    else 
        do! Log.error "Password change unauthorized: user %i" user.Id
        return Error "Old password is invalid"
}
```

Как можно видеть, параметр `env` больше не передается. Теперь это неявная часть выражения вычисления. Однако на данный момент в коде не предусмотрен набор операторов, которые бы заставили эту штуку работать.

Для начала понадобятся конструкторы `Effect<'env,'out>`:

``` fsharp
module Effect =
    /// Create value with no dependency requirements.
    let inline value (x: 'out): Effect<'env,'out> = Effect (fun _ -> x)
    /// Create value which uses depenendency.
    let inline apply (fn: 'env -> 'out): Effect<'env,'out> = Effect fn
```

Также нужно каким-то образом запустить `effect`, чтобы быть в состоянии сделать это... эффективно:

``` fsharp
module Effect =

    (* ...other functions... *)

    let run (env: 'env) (Effect fn): 'out = fn env
```

И поскольку ранее было упомянуто, что `Effect` - это монада, то также понадобится функция `bind`, для того была возможность скомпоновать эффекты вместе:

``` fsharp
module Effect =

    (* ...other functions... *)

    let inline bind (fn: 'a -> Effect<'env,'b>) effect =
        Effect (fun env ->
            let x = run env effect // compute result of the first effect
            run env (fn x) // run second effect, based on result of first one
        )
```

Почти закончили. Осталось скомпоновать всё в `EffectBuilder`, чтобы была возможность использовать выражение вычисления `effect { ... }`:

``` fsharp
[<Struct>]
type EffectBuilder =
    member inline __.Return value = Effect.value value
    member inline __.Zero () = Effect.value (Unchecked.defaultof<_>)
    member inline __.ReturnFrom (effect: Effect<'env, 'out>) = effect
    member inline __.Bind(effect, fn) = Effect.bind fn effect
    
let effect = EffectBuilder()
```

Теперь необходимо адаптировать модули, которые были реализованы ранее, чтобы они работали с "эффектами", а не с обычными функциями. Это легко сделать, используя функцию `Effect.apply`:

``` fsharp
module Log =    
    let debug fmt =
        let ap s = Effect.apply (fun (x: #ILog) -> x.Logger.Debug s)
        Printf.kprintf ap fmt
```

Итак, как можно заметить из окончательной версии функции `changePass` на основе "эффектов", в результате почти полностью удалён код подключения зависимостей. У этого подхода есть несколько недостатков:

- Необходимо сделать гораздо больше привязок (см. выражения `let!` / `do!`), что означает больше лямбда-замыканий, непрямых вызовов (можете посмотреть стек вызова) и больше выделения памяти.
- По-сути, также было удалено выражение вычисления `task { ... }`, а вместе с ним и готовый функционал для написания асинхронного кода. Это один из недостатков использования монад - композиция  - это боль.

Конечно, можно было бы улучшить тип `Effect`, чтобы была возможность работать с `Task` / `Async`. Однако, это значит, что паттерн усложняется и становится скорее фреймворком, а не чем-то, что можно легко применить к существующему коду. Плохо ли это? Не обязательно, но наверняка это потребует бОльших усилий, так как теперь необходимо писать не только бизнес-логику, но и заниматься поддержкой новой библиотеки. Возможно, в будущем эта концепция появится в экосистеме F#.

## Резюме

Начав с частичной реализации, в качестве инструмента для внедрения зависимостей, мы перешли к более структурированному подходу, в котором есть единый тип параметра `env` и закончили паттерном Монада Reader. Реализовать такое на практике - это непростой путь. Я надеюсь, что вы попробуете, и это поможет определить подход, который подходит именно вам.
